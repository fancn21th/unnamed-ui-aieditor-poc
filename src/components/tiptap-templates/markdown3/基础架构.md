# mardown3 基础架构

## 目标

mardown3 需要把两套能力合并到同一个编辑器中：

- 来自 markdown2 的基础编辑能力与工具栏交互
- 来自 markdown 的 Markdown 流式输出能力

最终结果是：编辑器既能手工编辑，也能持续接收外部流式 Markdown 内容并增量渲染。

## 目录结构

```text
mardown3/
├─ data/
│  └─ content.md
├─ mardown3-editor.scss
├─ mardown3-editor.tsx
└─ 基础架构.md
```

## 架构分层

### Layer 1：流输入层

- 输入来源可以是模拟流，也可以替换成真实 SSE/WebSocket/HTTP chunk 流。
- 当前通过 `createStreamSimulator` 将整段 Markdown 按随机 chunk 持续输出。

职责：只负责“按片段吐数据”，不感知编辑器结构。

### Layer 2：Markdown 节点缓冲层

- 使用 `MarkdownNodeBuffer` 聚合 Layer 1 的 chunk。
- 缓冲层按块级语义切分节点，并输出三种状态消息：
  - `start`
  - `updating`
  - `end`
- 对未闭合的 Markdown 片段进行修复，避免半截语法导致编辑器报错。

职责：把“字符串流”转换为“可被编辑器安全消费的节点消息流”。

### Layer 3：编辑器增量应用层

- 使用 `useStreamingEditor` 消费 Layer 2 消息。
- 通过 `insertContentAt(..., { contentType: "markdown" })` 增量写入。
- 逻辑策略：
  - `start`：在当前文档尾部插入起始内容并记录位置
  - `updating`：替换该节点的当前范围
  - `end`：以最终内容收敛该节点并清理状态

职责：把节点消息映射到 ProseMirror 文档变更。

## 编辑器层（UI + 扩展）

编辑器主体继承 markdown2 的交互模式：

- 顶部工具栏分组（撤销重做、标题/列表、行内样式、对齐、图片、主题）
- 移动端二级面板（highlighter/link）切换
- 光标可见性与工具栏浮动位置联动

扩展组合采用 Markdown 模式：

- `Markdown.configure({ markedOptions: { gfm: true } })`
- `contentType: "markdown"`
- 在此基础上叠加工具栏所需扩展（TaskList、Highlight、Subscript、Superscript、ImageUploadNode 等）

## 数据流

1. `content.md` 作为初始流文本源
2. Layer 1 持续产出 chunk
3. Layer 2 聚合并发出节点状态消息
4. Layer 3 将消息增量写入编辑器
5. 用户可在任何时刻继续手工编辑同一文档

## 当前约束

- 目前默认是“自动播放模拟流”，用于验证渲染链路稳定性。
- 若切真实业务流，建议把 Layer 1 抽象为可注入的 stream provider。
- 非标准 GFM 能力（如部分高亮样式）在纯 Markdown 交换时可能需要降级表达。

## 后续演进建议

- 增加“开始/暂停/重置流”控制按钮
- 支持外部传入 chunk 流回调接口
- 增加流式状态展示（idle/streaming/done/error）
- 增加回压与节流策略，避免大文档频繁重排
